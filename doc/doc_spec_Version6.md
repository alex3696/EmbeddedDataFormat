# EDF (Embedded Data Format) — спецификация

> **EDF** — компактный бинарно-текстовый блочный формат для хранения и обмена структурированными данными в embedded-системах с ограниченными ресурсами.  
> Формат создавался как альтернатива LAS и CSV, но с явным встроенным описанием схем (schema-on-data), поддержкой вложенных структур, хранением метаданных, надёжным восстановлением данных и минимальными требованиями к памяти.

## Введение и мотивация

**EDF** предназначен для экономичного хранения, надёжной записи и передачи структурированных бинарных (и текстовых) данных во flash/EEPROM микроконтроллеров, ведения логов и обмена между устройствами и ПК.  
Отличие от LAS, CSV и JSON:  
- Явная схема внутри данных (schema-on-data)
- Простота парсинга без внешнего описания структуры
- Лёгкое восстановление при потере/коррупции блока (даже без кэша схем на устройстве)
- Отсутствие динамического выделения памяти

---

## Область применения

- Логирование (телеметрия устройств, события)
- Хранение табличных данных во flash/EEPROM
- Обмен структурированными данными между embedded-устройствами и ПК
- Формат обмена/архивации для автономных и маломощных систем

## Главные принципы и ограничения

- **Блочность:** Всё хранится и передаётся блоками. Размер блока: 256…4096 байт.
- **Метаданные — внутри:** Схема данных сериализуется до данных.
- **Append-only:** Запись только с добавлением, идеально для флеш-памяти/журналов.
- **Без malloc:** Только статические и линейные аллокаторы.
- **Контроль целостности (CRC16 Modbus):** В бинарном режиме, по блоку.
- **Требуется Little Endian-архитектура.**
- **Строки:** Только UTF-8, максимум 255 байт (обрезка без предупреждения).
- **Структуры и вложенность ограничены по размеру блока — слишком сложные не сериализуются.**
- **Нет поддержки одновременного хранения и произвольной выборки данных разных схем: Data может относиться только к последней активной схеме (см. [Чтение множественных схем и записей](#чтение-множественных-схем-и-записей)).**

## Структура файла и блоков

### Таблица типов блоков

| Название   | Код (dec/hex) | Символ | Текстовый вид | Назначение             |
|------------|---------------|--------|---------------|------------------------|
| Конфиг     | 126 / 0x7E    | `~`    | `<~ ... >`    | Конфигурация файла     |
| Схема      | 63  / 0x3F    | `?`    | `<? ... >`    | Определение схемы      |
| Данные     | 61  / 0x3D    | `=`    | `<= ... >`    | Пользовательские данные|

---

### Общая структура блока (EdfBlock_t)

| Поле    | Размер    | Описание                               |
|---------|-----------|----------------------------------------|
| Type    | 1         | Тип блока                              |
| Len     | 2         | Длина Content                          |
| Content | [Len]     | Полезная нагрузка                      |
| CRC     | 2         | CRC16 (Modbus) от (Type+Len+Content)   |

> В текстовом режиме CRC отсутствует; конец блока отмечен символом `>`.

---

### Конфигурационный блок

**Первый блок в EDF-файле, задаёт глобальные параметры.**

| Поле      | Тип      | Описание                 |
|-----------|----------|--------------------------|
| VersMajor | uint8_t  | Главная версия формата   |
| VersMinor | uint8_t  | Минорная версия формата  |
| Encoding  | uint16_t | Кодировка (только 65001) |
| Blocksize | uint16_t | Размер блока (256–4096)  |
| Reserved  | uint16_t | Зарезервировано (0)      |
| Flags     | uint32_t | Зарезервировано (0)      |

**Пример:**
```
<~ {version=1.0; bs=512; encoding=65001; flags=0;} >
```

### Блок схемы (Schema–block)

**Содержит описание структуры пользовательских данных. Схема сериализуется полностью в один блок.**

#### Формат EdfSchema_t (бинарный):

| Поле       | Размер (байт)   | Описание                                            |
|------------|-----------------|-----------------------------------------------------|
| Id         | 2               | Идентификатор схемы (0…65535)                       |
| Name       | 1+N             | Имя схемы (byte length + bytes, ≤255)               |
| Desc       | 1+M             | Описание схемы (опционально, как строка)            |
| Type       | переменный      | Cостав поля (см. EdfType_t, рекурсивно)             |

#### Формат EdfType_t:

| Поле        | Размер (байт)    | Описание                                                        |
|-------------|------------------|-----------------------------------------------------------------|
| Type        | 1                | Prim/Struct, см. PoType                                         |
| DimsCount   | 1                | Кол-во измерений массива (0–255)                                |
| Dims        | DimsCount×2      | Размеры измерений (uint16_t)                                    |
| Name        | 1+N              | Имя типа/поля (строка)                                          |
| FieldsCount | 1 or отсутствует | Число полей (Struct) **Только если Type==Struct и FieldsCount>0**|
| Fields      | переменный       | (FieldsCount × EdfType_t)                                       |

**Примечания:**
- FieldsCount есть ТОЛЬКО если Type==Struct и FieldsCount>0
- Поля структуры хранятся строго в порядке объявления
- Максимум 255 полей в одной структуре (uint8_t)
- Общее число примитивов в структуре (включая вложенные и массивы) ≤ 65535

#### Полная таблица примитивных типов PoType

| Название | Код | Размер (байт) | C-тип      | Примечание                                       |
|----------|-----|---------------|------------|--------------------------------------------------|
| Struct   | 0   | –             | –          | Вложенная структура                              |
| Char     | 1   | 1             | char[N]    | Фиксированный массив (padded `\0`)               |
| Int8     | 2   | 1             | int8_t     |                                                  |
| UInt8    | 3   | 1             | uint8_t    |                                                  |
| Int16    | 4   | 2             | int16_t    |                                                  |
| UInt16   | 5   | 2             | uint16_t   |                                                  |
| Int32    | 6   | 4             | int32_t    |                                                  |
| UInt32   | 7   | 4             | uint32_t   |                                                  |
| Int64    | 8   | 8             | int64_t    |                                                  |
| UInt64   | 9   | 8             | uint64_t   |                                                  |
| Half     | 10  | 2             | uint16_t   | 16-bit float (IEEE 754-2008)                     |
| Single   | 11  | 4             | float      |                                                  |
| Double   | 12  | 8             | double     |                                                  |
| String   | 13  | 1+N           | char*      | Len (1 байт) + Data (≤255 байт, без '\0' в конце)|

**Пример бинарного блока схемы (кратко, LE):**
```
Id:       00 00
Name:     0A 'S' 'e' 'n' 's' 'o' 'r' 'D' 'a' 't' 'a'
Desc:     00
Type:     00 // Struct
Dims:     00
Name:     06 'S' 'e' 'n' 's' 'o' 'r'
Fields:   03
    Field0: 07 00 09 't' 'i' 'm' 'e' 's' 't' 'a' 'm' 'p'
    Field1: 0B 00 05 'v' 'a' 'l' 'u' 'e'
    Field2: 0D 00 04 'u' 'n' 'i' 't'
```

---

### Блок данных (Data–block)

Содержит данные пользователя, сериализованные по конкретной схеме.  

#### Формат EdfRecordContent_t (бинарный):

| Поле      | Размер            | Описание                                                            |
|-----------|-------------------|---------------------------------------------------------------------|
| SchId     | 2 (uint16_t)      | Идентификатор схемы, к которой относятся данные                     |
| PrmOffset | 2 (uint16_t)      | Смещение по примитивам в записи (для восстановления)                |
| RecId     | 4 (uint32_t)      | Логический номер записи (RecordId)                                  |
| Data      | перем.            | Сериализованные значения (см. схему: порядок, типы, вложенность)    |

> Примитив не может быть разорван между блоками, но элементы массива могут быть распределены между блоками.

**Пример (запись для схемы SensorData):**
```
00 00       // SchId
00 00       // PrmOffset
00 00 00 00 // RecId
80 96 AD 65 // timestamp = 1700000000 (UInt32)
00 00 BC 41 // value = 23.5 (float)
02          // len = 2 (строка)
D0 C2       // '°','C'
```

## Сериализация: бинарный и текстовый форматы

### Бинарный режим

- Все значения LE (Little Endian).
- Каждый блок завершается CRC16 (кроме текстового режима).
- Примитив — всегда целиком в блоке, массив — может быть "разделён".
- Сериализация строго по схеме: Fields структуры, затем их вложенность или элементы массива, и т.д.

### Текстовый режим

- Представление:  
  - `<~ ... >` — конфиг  
  - `<? ... >` — схема  
  - `<= {...}>` — данные
- Строки заключены в двойные кавычки, экранирование символов не поддерживается.
- Числа — в десятичном формате.
- Массивы, структуры — в фигурных скобках через `;` (row-major).
- CRC отсутствует, конец блока — символ `>`.

**Пример текстового файла:**
```
<~ {version=1.0; bs=512; encoding=65001; flags=0;} >

<? {0;"SensorData"} Struct {
  UInt32 "timestamp";
  Single "value";
  String "unit";
} >

<= {1700000000; 23.5;"°C";}
<= {1700000060; 23.6;"°C";}
```

## Чтение множественных схем и записей

**В EDF-файле разрешается размещать сколько угодно блоков схем (Schema), каждый из которых отделяет партию блоков данных (Data), относящихся к именно этой схеме.**  
После Schema идут блоки Data, **все они обязаны соответствовать последней объявленной схеме**; смешивать Data разных схем "вперемешку" нельзя, так как в минималистичной реализации не хранятся несколько схем в кэше.

**SchId** в блоке Data нужен только для восстановления после ошибок (например, пропуска или повреждения блока); любой Data-Chunk может быть ассоциирован только с текущей схемой.

> **Внимание!**  
> Между двумя блоками Schema все последующие блоки Data всегда принадлежат последней прочитанной схеме.  
> Смешивать записи разных типов в одном потоке нельзя.  
> В минимальной реализации схема не кэшируется. После появления нового блока Schema прежняя схема "забывается".

**Пример:**
```
<~ ... >
<? {0;"SensorData"} ... >
<= {...}>
<= {...}>
<? {1;"Pressure"} ... >
<= {...}>
<= {...}>
```

## API: функции работы с EDF

```c
// Инициализация
int EdfInit(EdfContext_t* ctx, uint8_t* mem, size_t memLen, const EdfConfig_t* cfg);
EdfContext_t* EdfCreate(uint8_t* mem, size_t memLen, const EdfConfig_t* cfg, int* pErr);

// Открытие потоков/файлов
int EdfOpenStream(EdfContext_t* ctx, Stream_t* stream, const char* mode);
int EdfOpenFile(EdfContext_t* ctx, const char* fname, const char* mode);

// Запись/чтение блоков, схем, данных
int EdfWriteConfig(EdfContext_t* ctx, size_t* writed);
int EdfWriteSchema(EdfContext_t* ctx, const EdfSchema_t* sch, size_t* writed);
int EdfWriteData(EdfContext_t* ctx, const void* data, size_t len);
int EdfFlushData(EdfContext_t* ctx, size_t* writed);

int EdfReadBlock(EdfContext_t* ctx);
int EdfReadBin(const EdfType_t* t, MemStream_t* src, LineAlloc_t* mem, void** presult, size_t* resultPrimOffset, size_t* primReaded);

int EdfClose(EdfContext_t* ctx);
```
- Для примитивов передавайте указатель на значение; для строк — на `const char*`.

## Примеры

```c
// Простейшая запись схемы и объекта
EdfConfig_t cfg = {0, 3, 65001, 512, 0, 0};
uint8_t mem[MEM_BLOCK_SIZE_512];

EdfContext_t* ctx = EdfCreate(mem, sizeof(mem), &cfg, &err);
```

## Практические ограничения, замечания и FAQ

- **Строки**: максимум 255 байт (UTF-8); лишнее обрезается.
- **Размер схемы и записи**: не может превышать Blocksize.
- **Struct**: ≤255 полей; всего ≤65535 примитивов включая вложенность.
- **LE only**; BE не поддерживается.
- **Отсутствие кэша схем**: ни одна Data-запись не может ссылаться на "старую" схему после появления новой Schema.
- **Нет произвольного смешивания типов данных**: все Data между Schema относятся к последней схеме.
- **Append-only**: не поддерживает удаление/модификацию существующих записей.

### FAQ

**Q:** Можно ли смешивать записи разных схем вперемежку (по SchId)?  
**A:** **Нет!** Data-блоки каждого типа идут строго после своей Schema (см. выше). Перемешивать Data для разных схем нельзя: схема не кэшируется. SchId в Data используется только для восстановления после потери блока.

**Q:** Что если строка длиннее 255 байт?  
**A:** Она тихо обрезается до 255 байт, ошибок и предупреждений не возникает.

**Q:** Как восстановить поток после потери блока?  
**A:** Используйте SchId, PrmOffset, RecId блоков Data для правильного пропуска и восстановления последовательности при повреждении/потере блока.

**Q:** Что произойдёт, если схема или запись не помещается в блок?  
**A:** Вернётся ошибка ERR_DST_SHORT или ERR_BLOCK_SIZE_LARGE.
